```
Read DECISIONS.md from the repo root and confirm the highest decision ID
before acting on anything else. If D-137 and D-138 are not present, stop
and tell me — do not proceed.

CONTEXT:
CLIVE v0.8 has added a tool_registry table (see
infrastructure/sql/init/07_tool_registry.sql). Block 13 is the Central
Orchestrator / Event Bus. It currently dispatches action events without
checking whether the tool is registered or enabled. Your task adds a
validation gate and the admin enable/disable event handlers.

YOUR TASK — Two changes to src/orchestrator/:

1. REGISTRY VALIDATION GATE
   Before dispatching any action event (events that result in Block 9
   execution):

   - Query clive_state.tool_registry WHERE tool_name = <requested_tool>
   - If no row found: do NOT dispatch. Emit action.rejected:
     { "tool_name": <name>, "reason": "tool_not_registered",
       "original_event_id": <id> }
   - If row found but enabled = FALSE: do NOT dispatch. Emit action.rejected:
     { "tool_name": <name>, "reason": "tool_disabled",
       "original_event_id": <id> }
   - If row found but deprecated = TRUE: do NOT dispatch. Emit action.rejected:
     { "tool_name": <name>, "reason": "tool_deprecated",
       "original_event_id": <id> }
   - If row found, enabled = TRUE, deprecated = FALSE: dispatch normally.

   Log all rejections at WARN level with the reason and original_event_id.

   Read src/orchestrator/ to identify which event types correspond to
   Block 9 actions. Do not hardcode new event type names — the gate
   should work by inspecting the tool_name field in the action event
   payload.

2. ADMIN TOOL ENABLE/DISABLE EVENT HANDLERS
   Block 13 must handle two new events emitted by Block 23:

   a. admin.tool_disable  (payload: { "tool_name": <name>, "confirmed": true })
      - UPDATE clive_state.tool_registry
        SET enabled = FALSE, updated_at = NOW()
        WHERE tool_name = <name>
      - If updated: emit admin.tool_updated
        { "tool_name": <name>, "action": "disabled" }
      - If tool_name not found: emit admin.tool_error
        { "tool_name": <name>, "reason": "tool_not_found" }

   b. admin.tool_enable  (payload: { "tool_name": <name>, "confirmed": true })
      - UPDATE clive_state.tool_registry
        SET enabled = TRUE, updated_at = NOW()
        WHERE tool_name = <name>
      - If updated: emit admin.tool_updated
        { "tool_name": <name>, "action": "enabled" }
      - If tool_name not found: emit admin.tool_error
        { "tool_name": <name>, "reason": "tool_not_found" }

READ FIRST (before making any changes):
- Read src/orchestrator/ in full
- Read infrastructure/sql/init/07_tool_registry.sql (exact schema)
- Read docs/spec/Block 13 - Orchestrator Source Code Specification.md

CONSTRAINTS:
- D-003: the registry check is a DB read within Block 13's own execution,
  not a cross-block call. This is acceptable.
- D-025: at-least-once delivery applies to events that pass the gate.
  The rejection path emits a new event; it is not a retry.
- D-006: the enable/disable events must arrive with confirmed: true.
  Block 13 trusts the confirmed flag — it does not re-confirm.
  Block 23 owns the confirmation flow (separate session).
- Use the existing DB connection pool; do not add a new one.

DONE WHEN:
- Block 13 rejects action events for unregistered/disabled/deprecated tools
- Rejections emit action.rejected (not silent drops)
- admin.tool_disable and admin.tool_enable events are handled
- Existing tests pass
- New unit tests cover: rejection for unregistered, rejection for disabled,
  successful enable, successful disable, tool_not_found on enable/disable
- Changes committed to main
```
